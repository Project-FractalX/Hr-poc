# 🎭 Saga Orchestrator: fractalx-saga-orchestrator

Auto-generated by **FractalX**. This service manages distributed sagas using the **Orchestrator Saga Pattern** to coordinate multi-step transactions across microservices.

## 1. What This Service Does

The saga orchestrator ensures data consistency across services that cannot share a single database transaction. It:

- Coordinates ordered remote calls across services via **NetScope gRPC**.
- Persists each execution's state in the `saga_instance` table — so progress survives restarts.
- Automatically triggers **compensating transactions** in **reverse order** if any step fails.
- Exposes REST endpoints to **start**, **query**, and **list** saga executions.

## 3. Saga State Machine

Each saga execution transitions through the following states:

```
STARTED → IN_PROGRESS → DONE
                  ↓
            COMPENSATING → FAILED
```

| Status | Meaning |
| :--- | :--- |
| `STARTED` | A new saga instance was created and saved. |
| `IN_PROGRESS` | Forward steps are executing. |
| `DONE` | All steps completed successfully. |
| `COMPENSATING` | A step failed; compensating transactions are running in reverse. |
| `FAILED` | All compensations have run; the saga is marked permanently failed. |

## 6. Database: `saga_instance` Table

This service uses an **H2 in-memory database** (default). Saga execution state is stored in a single table:

| Column | Type | Description |
| :--- | :--- | :--- |
| `id` | `BIGINT IDENTITY` | Auto-generated primary key |
| `saga_id` | `VARCHAR(100)` | Saga type identifier (e.g. `place-order-saga`) |
| `correlation_id` | `VARCHAR(36)` | Unique execution UUID |
| `owner_service` | `VARCHAR(100)` | Service that triggered this saga |
| `status` | `VARCHAR(30)` | Current `SagaStatus` value |
| `current_step` | `VARCHAR(200)` | Last attempted step (`service:method`) |
| `payload` | `TEXT` | Original JSON payload |
| `error_message` | `TEXT` | Failure message if any |

Migration script: `src/main/resources/db/migration/V1__init_saga.sql`

> **⚠️ Note:** Data is lost when this service stops (H2 in-memory).
> For production, configure a persistent datasource — see **Configuration Guide** below.
> Access H2 Console: `http://localhost:8099/h2-console` (JDBC URL: `jdbc:h2:mem:saga_db`)

## 7. Service Connectivity

The orchestrator reaches participating services via **NetScope gRPC**. Default ports (service port + 10 000):

| Service | HTTP Port | gRPC Port |
| :--- | :---: | :---: |
| `department-service` | 8085 | 18085 |
| `employee-service` | 8081 | 18081 |
| `leave-service` | 8083 | 18083 |
| `payroll-service` | 8082 | 18082 |
| `recruitment-service` | 8084 | 18084 |

Configured in `application.yml` under `netscope.client.servers`.

## 8. Configuration Guide

To persist saga state across restarts, replace the H2 datasource in `application.yml`:

```yaml
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/saga_db
    username: postgres
    password: <your_password>
    driver-class-name: org.postgresql.Driver
  jpa:
    hibernate:
      ddl-auto: validate
```

Add the PostgreSQL driver to `pom.xml`:

```xml
<dependency>
    <groupId>org.postgresql</groupId>
    <artifactId>postgresql</artifactId>
</dependency>
```
